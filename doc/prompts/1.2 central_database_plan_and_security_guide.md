# Central Database Setup Plan & Security Guide for oU1TS

This document outlines a step-by-step plan to establish a centralized database for the **oU1TS Campus Digital Ecosystem** and provides a guide on standard security measures to protect database secrets.

---

## 1. Central Database Setup Plan

Since the projects (Portal, Archive, and Scheduler) are migrating to subdirectories under the main domain `https://ou1ts.github.io`, a single, central Supabase project is the most efficient and scalable solution, especially within Supabase's Free Tier limits.

### Step 1: Initialize Database Schema in Supabase

Run the following SQL script in your Supabase SQL Editor to set up the core `profiles` and `stars` tables.

```sql
-- =========================================================
-- 1. PROFILES TABLE (Shared across all sub-projects)
-- =========================================================
CREATE TABLE public.profiles (
    id UUID REFERENCES auth.users ON DELETE CASCADE PRIMARY KEY,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL,
    email TEXT UNIQUE NOT NULL,
    student_id TEXT UNIQUE,
    full_name TEXT,
    department TEXT,
    -- Tracks which sub-projects the user interacts with or has access to
    project_tags TEXT[] DEFAULT ARRAY['root']::TEXT[] NOT NULL
);

-- Enable Row Level Security (RLS)
ALTER TABLE public.profiles ENABLE ROW LEVEL SECURITY;

-- =========================================================
-- 2. STARS TABLE (User favorites/interactions)
-- =========================================================
CREATE TABLE public.stars (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id UUID REFERENCES public.profiles(id) ON DELETE CASCADE NOT NULL,
    resource_type TEXT NOT NULL, -- e.g., 'project', 'course', 'link'
    resource_id TEXT NOT NULL,   -- The unique identifier of the starred item
    created_at TIMESTAMP WITH TIME ZONE DEFAULT timezone('utc'::text, now()) NOT NULL,
    
    -- Ensure a user can only star a specific resource once
    CONSTRAINT unique_user_resource_star UNIQUE (user_id, resource_type, resource_id)
);

-- Enable Row Level Security (RLS)
ALTER TABLE public.stars ENABLE ROW LEVEL SECURITY;
```

### Step 2: Configure PostgreSQL Triggers for New Sign-ups

To guarantee that every user signed up through *any* sub-project gets a central profile entry automatically, set up this trigger function:

```sql
-- Trigger function to automatically create a profile on user sign-up
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO public.profiles (id, email, student_id, project_tags)
    VALUES (
        NEW.id,
        NEW.email,
        COALESCE(NEW.raw_user_meta_data->>'student_id', 'OAUTH_USER'),
        ARRAY['root']::TEXT[]
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Bind trigger function to auth.users
CREATE OR REPLACE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

### Step 3: Apply Row Level Security (RLS) Policies

To secure user profile data and starring information directly at the database level:

```sql
-- Profiles RLS Policies
CREATE POLICY "Public profiles are viewable by everyone" 
ON public.profiles FOR SELECT 
USING (true);

CREATE POLICY "Users can update their own profile" 
ON public.profiles FOR UPDATE 
USING (auth.uid() = id);

-- Stars RLS Policies
CREATE POLICY "Stars are viewable by everyone" 
ON public.stars FOR SELECT 
USING (true);

CREATE POLICY "Authenticated users can create stars for themselves" 
ON public.stars FOR INSERT 
WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can delete their own stars" 
ON public.stars FOR DELETE 
USING (auth.uid() = user_id);
```

## 2. Security Guide: Protecting Database Secrets

### Core Security Principle
In modern serverless architectures like Supabase, there are two distinct keys:
1. **Anonymous (`anon`) Key**: Safe to expose to the client. It respects Row-Level Security (RLS) policies.
2. **Service Role (`service_role`) Key**: **Never expose to the client.** It bypasses RLS policies completely and has administrative access to the database.

> [!WARNING]
> Hardcoding any API key inside frontend files (like `supabase-config.js`) makes it easy to accidentally check the credentials into public GitHub repositories. If a public repository contains your credentials, scanners can steal your keys.

---

### Measure A: Securing Keys in Pure Frontend (Build-Time Variables)

If the sub-projects remain strictly frontend-only (hosted on GitHub Pages), you cannot hide the keys from the browser's Network tab. However, you should prevent them from being hardcoded in your source code.

#### 1. Move Keys to Environment Files
Create a `.env` file in the root of each sub-project:
```env
VITE_SUPABASE_URL=https://your-project-id.supabase.co
VITE_SUPABASE_ANON_KEY=your-client-anon-key
```
Add `.env` (and `.env.local`, `.env.production`) to your `.gitignore` file so they are never committed.

#### 2. Reference in Code
Modify `supabase-config.js` to reference the environment variables dynamically:
```javascript
const SUPABASE_URL = import.meta.env.VITE_SUPABASE_URL;
const SUPABASE_ANON_KEY = import.meta.env.VITE_SUPABASE_ANON_KEY;

window.supabaseClient = supabase.createClient(SUPABASE_URL, SUPABASE_ANON_KEY);
```

#### 3. Set Up Secrets in GitHub Actions
When building and deploying to GitHub Pages via CI/CD, add your secrets in **GitHub > Repository Settings > Secrets and Variables > Actions** as:
- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

In your workflow YAML, inject them during build:
```yaml
- name: Build
  run: npm run build
  env:
    VITE_SUPABASE_URL: ${{ secrets.VITE_SUPABASE_URL }}
    VITE_SUPABASE_ANON_KEY: ${{ secrets.VITE_SUPABASE_ANON_KEY }}
```

---

### Measure B: Hiding Client Keys via a Serverless Proxy Backend

If you want to prevent users from seeing the Supabase project URL and anon key entirely (e.g., to prevent external scraping of database endpoints), you must route requests through a backend. 

```
┌────────────────┐     HTTP Requests     ┌────────────────┐    Admin Actions    ┌────────────────┐
│  Client App    │ ────────────────────> │  Backend Proxy │ ──────────────────> │  Supabase DB   │
│ (GitHub Pages) │ <──────────────────── │ (Vercel/Node)  │ <────────────────── │  (Cloud Host)  │
└────────────────┘     JSON Responses    └────────────────┘    Service Role     └────────────────┘
```

#### Architecture Plan
1. **Create a separate backend repository** (e.g., `ou1ts-api`) deployed to a platform like Vercel, Netlify, or Render.
2. **Store secrets** (`SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` or `SUPABASE_ANON_KEY`) strictly in the hosting platform's environment variables.
3. **Write endpoint handlers** to proxy standard auth and database queries.

#### Example Backend Implementation (NodeJS / Express)

Here is a template for a server-side route that proxies user stars securely without exposing any Supabase keys to the browser:

```javascript
// server.js (NodeJS Backend API)
const express = require('express');
const cors = require('cors');
const { createClient } = require('@supabase/supabase-client');

const app = express();
app.use(express.json());

// Restrict CORS to oU1TS frontend domains only
const allowedOrigins = ['https://ou1ts.github.io', 'http://localhost:5173'];
app.use(cors({
    origin: function(origin, callback) {
        if (!origin || allowedOrigins.indexOf(origin) !== -1) {
            callback(null, true);
        } else {
            callback(new Error('Blocked by CORS policy'));
        }
    }
}));

// Initialize Supabase admin client using server-side environment variables
const supabase = createClient(
    process.env.SUPABASE_URL,
    process.env.SUPABASE_SERVICE_ROLE_KEY // Kept completely hidden on the server
);

// Proxy Route: Toggle Star
app.post('/api/stars/toggle', async (req, res) => {
    // Authenticate the incoming request using the client's bearer token
    const authHeader = req.headers.authorization;
    if (!authHeader) return res.status(401).json({ error: 'Access denied' });

    const token = authHeader.split(' ')[1];
    
    // Validate token against Supabase Auth
    const { data: { user }, error: authError } = await supabase.auth.getUser(token);
    if (authError || !user) return res.status(401).json({ error: 'Invalid session' });

    const { resourceId, resourceType, isStarred } = req.body;

    try {
        if (isStarred) {
            // Delete star
            const { error } = await supabase
                .from('stars')
                .delete()
                .eq('user_id', user.id)
                .eq('resource_type', resourceType)
                .eq('resource_id', resourceId);

            if (error) throw error;
            return res.json({ success: true, action: 'unstarred' });
        } else {
            // Insert star
            const { error } = await supabase
                .from('stars')
                .insert({
                    user_id: user.id,
                    resource_type: resourceType,
                    resource_id: resourceId
                });

            if (error) throw error;
            return res.json({ success: true, action: 'starred' });
        }
    } catch (err) {
        return res.status(500).json({ error: err.message });
    }
});

const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Secure backend running on port ${PORT}`));
```
